Welcome to the "Angular Enterprise Architecture" example. The application showcases many of the architectural approaches discussed in the ebook and can serve as an example (or starting point) when implementing your own project.

Focus on architecture

The example focuses on architecture and its automated validation and comes with an example for ech individual concept as well as all eslint rules out of the box. Adding some forbidden import, for example, try to import ProductService in the OrderDashboardComponent and run ng lint to see it in action.

  • Architecture example - the project showcases all concepts that have been discussed in the ebook, namely core, ui, pattern, feature and shared-feature and how they can be used together. The most of real life applications can almost always be expressed as a unique combination of these concepts.
  • Automated Architecture validation - the architecture validation is automated with the help of eslint-module-boundaries plugin which can guarantee that the architecture stays clean through out the project life time instead of situation where folder structure looks clean but hides a convoluted dep graph behind the scenes.
  • Modern Standalone APIs - the setup uses the latest modern Standalone APIs and approaches, see example in core/core.ts in form of provideCore() as an inspiration how to implement your own functionality in a standalone compatible way.
  • Extendable - the setup supports extending of the project by creating additional apps and libs in this workspace. The provided rules for architecture validation cover this scenario as well.

Decisions to be made

The example focuses on architecture and related concerns and is therefore not opinionated about many topics that need to be handled by most projects but are at the same time quite unique to every project and organization.

  • State management - depending on the amount of state, the need to share state between multiple features and the complexity of how state has to change based on user interaction, the app will land somewhere on the spectrum from component based state management to full blown state management library like @ngrx/store
  • Component library - the example comes with @angular/material out of the box but there are many other 3rd party component libraries like primeng, Taiga UI, or even custom...
  • SSR/ SSG - is the app a public website that needs to be indexed by the search engines? In that case, SSR or SSG can dramatically improve SEO, Lighthouse score and performance.
  • I18n - build time ($localize), runtime (3rd party libs like ngx-translate), or none
  • Authentication & Authorization - does app even need to manage it?, SSO, OAuth, 3rd party...
  • Other - every project is unique with different set of requirements and constrains...
